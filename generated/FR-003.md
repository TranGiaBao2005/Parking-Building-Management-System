### Business Rules & Constraints

1. **Slot Status Transitions**
   - VACANT ↔ IN_USE: Entry/exit cycle
   - VACANT → PRE_BOOKED: Reservation made
   - IN_USE → MAINTENANCE: Mark for repairs while occupied (rare)
   - any → BLOCKED: Emergency closure
   - MAINTENANCE ↔ VACANT: Maintenance complete
   - PRE_BOOKED → IN_USE: Reservation activated
   - PRE_BOOKED → VACANT: Reservation expired/cancelled

2. **Slot Code Format**
   - Format: [SECTION_CODE]-[NUMBER], e.g., A-001, B-047
   - Must be unique per facility
   - Used in customer tickets and reports

3. **Concurrent Access Control**
   - Slot reservation must be transactional
   - Prevent double-booking same slot
   - Use pessimistic locking (row-level LOCK) or optimistic (version checking)

4. **Availability Queries**
   - Query vacant slots in section: response ≤500ms (use index + cache)
   - Availability snapshot: refresh every 5 minutes or on slot change
   - Include occupancy percentage

## SECTION 3: BACK-END SPECIFICATIONS

### Database Schema

```sql
CREATE TABLE parking_slots (
    slot_id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    facility_id UUID NOT NULL REFERENCES facilities(facility_id),
    section_id UUID NOT NULL REFERENCES facility_sections(section_id),
    slot_code VARCHAR(20) NOT NULL,
    floor_number INTEGER NOT NULL,
    status VARCHAR(20) NOT NULL DEFAULT 'VACANT'
        CHECK (status IN ('VACANT', 'IN_USE', 'PRE_BOOKED', 'MAINTENANCE', 'BLOCKED')),
    is_handicap_accessible BOOLEAN DEFAULT false,
    last_maintenance_at TIMESTAMP,
    notes VARCHAR(500),
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP AT TIME ZONE 'UTC',
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP AT TIME ZONE 'UTC',
    UNIQUE(facility_id, slot_code),
    FOREIGN KEY (section_id) REFERENCES facility_sections(section_id)
);

CREATE INDEX idx_slots_facility_status ON parking_slots(facility_id, status);
CREATE INDEX idx_slots_section_status ON parking_slots(section_id, status);
CREATE INDEX idx_slots_code ON parking_slots(facility_id, slot_code);
```

### Exception Handling

| Scenario | HTTP Status | Response |
|----------|------------|----------|
| Concurrent slot reservation | 409 | `{"error": "Slot already reserved"}` |
| Invalid status transition | 400 | `{"error": "Cannot transition from IN_USE to PRE_BOOKED"}` |
| Duplicate slot code | 409 | `{"error": "Slot code already exists"}` |
| Slot not found | 404 | `{"error": "Slot not found"}` |
| Bulk create exceeds section capacity | 400 | `{"error": "Bulk creation would exceed section capacity"}` |
| Bulk creation database error | 500 | Transactional rollback, entire operation fails |

## SECTION 4: ACCEPTANCE CRITERIA

### Scenario 1: Create Multiple Slots Atomically (Happy Path)
```gherkin
Given Manager accessing Section A-01 (capacity 150)
When Manager bulk creates 150 slots with 5 handicap-accessible
Then system creates all 150 slots in single transaction
And slots numbered A-001 through A-150
And 5 slots marked is_handicap_accessible=true
And returns HTTP 201
And all slots appear in slot list
```

### Scenario 2: Update Slot from Vacant to In-Use (Happy Path)
```gherkin
Given slot A-001 with status VACANT
When staff updates status to IN_USE
Then system transitions slot status
And updates slot.updated_at timestamp
And returns HTTP 200
And subsequent queries show slot as occupied
```

### Scenario 3: Prevent Invalid Status Transition (Edge Case)
```gherkin
Given slot with status IN_USE
When staff attempts to update to PRE_BOOKED
Then system rejects transition
And returns HTTP 400
And status remains IN_USE
And error message: "Cannot transition from IN_USE to PRE_BOOKED"
```

### Scenario 4: Handle Concurrent Slot Reservation (Edge Case)
```gherkin
Given slot A-001 with status VACANT
When two concurrent entry requests attempt to reserve same slot
Then system locks slot (pessimistic) or uses versioning (optimistic)
And only one transaction succeeds
And other returns HTTP 409: "Slot already reserved"
And exactly one parking session created
```

### Scenario 5: Get Availability with Sub-500ms Response (Performance)
```gherkin
Given facility with 500 slots, 287 occupied
When driver queries /api/v1/facilities/:id/availability
Then system returns within 500ms
And includes occupancy by section
And data freshness ≤5 minutes
And returns HTTP 200
```
